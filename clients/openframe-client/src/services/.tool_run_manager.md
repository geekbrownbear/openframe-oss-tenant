Manages tool execution processes with support for both cross-platform and Windows-specific user session handling. Provides robust process management including retry logic, kill integration, and per-user-session lifecycle handoff on Windows.

## Key Components

- **ToolRunManager** - Main service for executing tools with comprehensive process management. Holds an optional `Arc<WindowsSessionManager>` — present only when running under the Windows SCM (service mode), `None` in direct/foreground mode.
- **Process Lifecycle** - Handles tool startup, monitoring, and cleanup with retry mechanisms (every `RETRY_DELAY_SECONDS`).
- **Cross-platform Support** - Standard `tokio::process::Command` spawn for Unix and Windows direct-mode; macOS uses a `launchctl`-based user session helper.

### Core Functions

- `get_active_user_session()` - Detects active Windows user sessions for GUI tool execution. Prefers RDP sessions over Console.
- `launch_process_in_user_session(command_path, args)` - Resolves the active user session and delegates to `launch_process_in_target_session`.
- `launch_process_in_target_session(command_path, args, session_id)` - Wraps `WTSQueryUserToken` + `DuplicateTokenEx` + `CreateProcessAsUserW` to launch a process in a specific session's user context. Used by `WindowsSessionManager`.
- `launch_process_in_console_session()` - Legacy alternative Windows process launching method.
- `run_tool` - Per-tool runner. For `Installation::GuiApp` on Windows in service mode, hands the lifecycle off to `WindowsSessionManager::register_tool`; in direct mode, falls through to the standard `Command::new` spawn.

## Usage Example

```rust
use crate::services::tool_run_manager::ToolRunManager;
use crate::models::installed_tool::InstalledTool;

// Direct/foreground mode — no SCM session manager
let tool_manager = ToolRunManager::new(
    installed_tools_service,
    params_resolver,
    kill_service,
    #[cfg(target_os = "windows")] None,
);

// On Windows service mode, ToolRunManager.new is called with the live
// `Arc<WindowsSessionManager>` so GuiApps get per-session waiters.

tool_manager.run().await?; // launches all installed tools
```

The manager handles Windows complexities like WTS session enumeration, user token duplication, and proper desktop assignment via the `launch_process_in_target_session` helper, which is also re-used by `WindowsSessionManager` for per-session launches.
