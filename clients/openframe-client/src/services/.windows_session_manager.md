Windows-only service that tracks active user sessions reported by the Windows SCM and keeps registered GuiApp tools running in each of them. Used by `ToolRunManager` when the client is running as a Windows service (SYSTEM in session 0) and needs to launch and supervise per-user GUI processes.

## Key Components

- **`WindowsSessionManager`** - Central registry of tools and active sessions. Spawned waiter tasks find or launch each registered tool in each active session and restart it if it exits.
- **`SessionEvent`** - Internal representation of `windows_service::ServiceControl::SessionChange` reasons (Logon, Logoff, ConsoleConnect/Disconnect, RemoteConnect/Disconnect).
- **`new(session_rx)`** - Builds the manager, seeds active sessions from a one-time WTS enumeration, and spawns the event-dispatch task that consumes events from the SCM control handler.
- **`register_tool(tool_id, command_path, launch_args, new_tool)`** - Idempotently registers a GuiApp and ensures a waiter exists for every active session. Re-registering updates the stored args in place. The `new_tool` boolean controls the startup wait behavior: when `false`, waiters delay `SESSION_STARTUP_DELAY_SECONDS` before attempting to find/launch processes, allowing Windows autorun entries time to launch; when `true`, waiters start immediately.
- **`waiter_task`** - One spawned task per `(tool_id, session_id)`. Loops: `find pid → attach OR launch → wait → restart` until the session goes inactive or the tool is unregistered.
- **`enumerate_active_user_sessions()`** - One-shot WTS scan returning session IDs in the `WTSActive` state, excluding session 0 and listen sessions.
- **`find_pid_in_target_session(exe_path, session_id)`** - Locates a running process matching `exe_path` (by full image path) in a given session, used to attach to processes Windows already launched via HKLM Run at logon.

## Usage Example

```rust
use tokio::sync::mpsc;
use crate::services::windows_session_manager::{WindowsSessionManager, SessionEvent};

// In the Windows service main: the SCM control handler sends events into the channel,
// and the manager owns the receiver.
let (session_tx, session_rx) = mpsc::unbounded_channel::<SessionEvent>();

let manager = WindowsSessionManager::new(session_rx);

// Register a GuiApp tool. Waiters are spawned for every currently-active session;
// future Logon events spawn new waiters automatically.
manager.register_tool(
    "openframe-chat".to_string(),
    r"C:\ProgramData\OpenFrame\tools\openframe-chat\openframe-chat.exe".to_string(),
    vec!["--background".to_string()],
    false, // new_tool: wait for autorun to launch before attaching
).await;
```

The manager pairs with an HKLM `Software\Microsoft\Windows\CurrentVersion\Run` entry written at install time (see `utils::windows_helpers::register_autorun`): Windows auto-launches the app at logon, the manager observes the existing PID via `find_pid_in_target_session` and attaches a waiter rather than double-launching. New logons after the service starts are picked up via SCM `SessionChange` events.

**Fallback-launch race condition:** Although the manager prefers to attach to an existing autorun process via `find_pid_in_target_session` and SCM `SessionChange` events, it only waits a bounded startup window (`SESSION_STARTUP_DELAY_SECONDS`, currently 5 minutes) before invoking the autorun entry itself. On slow logons or when Windows delays running the autorun entry, this can result in a duplicate process being launched. The trade-off balances reliability (ensuring the tool runs even if autorun fails) against the small risk of duplication. To adjust this behavior, modify `SESSION_STARTUP_DELAY_SECONDS` in `windows_session_manager.rs` or the `wait_for_autorun` logic in the `ensure_waiter` and `waiter_task` functions.
