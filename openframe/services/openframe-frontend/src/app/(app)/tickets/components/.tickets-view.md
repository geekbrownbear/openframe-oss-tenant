Main view component for the tickets interface that manages tab navigation and URL parameter state synchronization.

## Key Components

- **`TicketsView`** - Primary component that orchestrates the tickets interface layout and state management
- **`useApiParams`** - Hook for managing URL parameters with type safety and default values
- **`TicketsTabNavigation`** - Child component for tab switching UI
- **`TicketsTabContent`** - Child component that renders content based on active tab and filters

## Usage Example

```typescript
import { TicketsView } from './tickets-view'

// Use in a tickets page or dashboard
export default function TicketsPage() {
  return (
    <div className="container mx-auto p-6">
      <h1 className="text-2xl font-bold mb-4">Support Tickets</h1>
      <TicketsView />
    </div>
  )
}
```

The component automatically synchronizes UI state with URL parameters, allowing users to bookmark and share specific ticket views. The `tab` parameter defaults to 'current' and `status` maintains an array of filter values, both persisted in the URL for seamless navigation and state management across page refreshes.

## Empty State

All three view implementations — `TicketsTable` (table, also used by the archive page via `CurrentTickets`/`ArchivedTickets`), `TicketsBoard`, and `TicketsBoardLifecycle` — render the shared `EmptyState` (over the library `NoData`) as a child of `PageLayout`, keeping the "Search for Ticket" bar visible above it. It is the simple message variant: a tag icon, "Ticket history empty", and "Tickets will appear here when available" (no feature blocks or button). It shows only when there is genuinely no data: not loading, no active search, no active filters (status for the table; organization/assignee for the boards), and zero tickets (all board columns empty). An active search or filter that returns nothing still shows the inline "No tickets found" message instead.
