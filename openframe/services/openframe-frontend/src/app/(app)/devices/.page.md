A client-side Next.js page component that renders the devices management interface with dynamic content loading and proper layout structure.

## Key Components

- **Dynamic Export Configuration** - Forces dynamic rendering to ensure fresh data on each request
- **Devices Component** - Main page component that wraps the devices view in the application layout
- **Suspense Boundary** - Provides loading state while the DevicesView component loads
- **AppLayout Integration** - Uses the shared application layout wrapper for consistent UI structure

## Usage Example

```typescript
// This page is automatically routed by Next.js App Router
// Accessible at /devices route

// The component structure:
<Suspense fallback={<LoadingSpinner />}>
  <AppLayout>
    <DevicesView />
  </AppLayout>
</Suspense>

// Dynamic configuration ensures:
// - No static generation
// - Server-side rendering on each request
// - Fresh device data on page load
```

The page leverages Next.js App Router conventions with the `'use client'` directive for client-side interactivity, while the `dynamic = 'force-dynamic'` export ensures the devices data remains current by preventing static generation. It renders `DevicesPanel` directly (no Suspense wrapper), passing the empty state via the `emptyState` prop.

## Empty State

The page passes an onboarding empty state to `DevicesPanel` via its `emptyState` prop — a shared `EmptyState` (over the library `NoData`) with the "No devices connected" heading, a description, three feature-highlight blocks (monitor health, run scripts, remote sessions), and an "Ask Mingo about Devices" button. `DevicesPanel` renders it (centered, as a direct child of `PageLayout`, with no filter toolbar or table/grid) only when the view is genuinely empty: loading finished, no error, no active search, no active filters (statuses/OS/organization/tags), and zero devices. A search or filter that returns no matches still shows the inline "No devices found. Try adjusting your search or filters." message. The prop is intentionally omitted in embedded usages (e.g. the customer details Devices tab), so the global onboarding copy does not leak there.
